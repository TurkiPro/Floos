# Implementation Plans

Generated by the improve skill. Batch 1 (001–004) and batch 2 (005–008) are
DONE and on main. **Batch 3 (009–019)** was generated on 2026-07-15 against
commit `db51b40` by a deep audit (9 parallel category auditors over every line
of `lib/`, tests, and platform config, each finding adversarially verified,
then re-vetted by the advisor against the source). Execute in the order below
unless dependencies say otherwise. Each executor: read the plan fully before
starting, honour its STOP conditions, and update your row when done.

The codebase remains in good shape — domain math is clean and well-tested, the
data layer is careful, CI is real. Batch 3 is dominated by two themes: **the
privacy promise has three holes** (cloud backup of the DB, runtime font
fetching, a lock that only covers Home), and **derived numbers go quietly
stale or wrong at the edges** (today's spending excluded from the weekly
budget, alerts not re-armed after mutations).

## Findings vetted but not planned (fix opportunistically or on request)

Bugs / correctness, in rough priority order:

- **Archiving a parent category strands its active sub-categories**
  (`category_editor_screen.dart:127-139`, `category_picker.dart:45-61`): the
  tree and picker only show children under *active* tops, so subs of an
  archived parent become invisible, unpickable, and un-archivable — yet stay
  "active". Fix sketch (S): `CategoryDao.archive` archives children in the
  same transaction; confirm dialog copy mentions it.
- **Income screen shows income filtered out of a 200-row mixed stream**
  (`income_screen.dart:121-125`): `watchRecent()` caps at 200 rows of *all*
  types, then filters income — heavy spenders see income history silently
  truncated. Fix sketch (S): a `watchRecentByType(TxnType)` DAO query.
- **Recurrence sheet permits endDate < startDate after editing startDate**
  (`add_recurrence_sheet.dart:243-279`): the end-date picker's
  `initialDate: _endDate ?? _startDate` with `firstDate: _startDate` can also
  assert-crash when `_endDate` predates a newly-moved `_startDate`. Fix
  sketch (S): clamp `_endDate` when `_startDate` moves; validate in `_save`.
- **Schedule edits/reactivation skip an occurrence due today**
  (`database.dart:280,324`): the marker is set to `dateOnly(now)` and
  catch-up generates strictly *after* it — reactivating a salary rule on
  payday silently drops today's salary. Fix sketch (S): set the marker to
  yesterday; decide whether that's wanted for reactivation too (it changes
  the documented "never backfill" nuance by exactly one day).
- **Timezone-lookup failure silently disables all notifications forever**
  (`notification_service.dart:29-47`): any `init()` throw leaves
  `_ready=false` with no retry and no user-visible signal. Fix sketch (S):
  fall back to `tz.local` default on lookup failure; only hard-disable on
  plugin init failure.
- **Weekly-budget repeating notification bakes a stale figure**
  (`notification_service.dart:139-151`): the money amount in the body is
  computed at schedule time and repeats weekly unchanged if the app isn't
  opened. Fix sketch (S): schedule a one-off for the next Saturday instead
  (re-armed each launch, like everyOtherDay), trading "fires even if the app
  is never opened" for a correct number.
- **`TZDateTime.add(Duration)` drifts an hour across DST**
  (`notification_service.dart:237,245`): reminder hour shifts for users in
  DST countries (e.g. Egypt) until the next reschedule. Fix sketch (S): step
  by calendar components (`TZDateTime(tz.local, y, m, d+1, h, min)`).
- **Future-dated transactions inflate month projection / daily average /
  no-spend-days** (`statistics_summary.dart:146-150,185`): arguably intended
  (planned expenses); decide and document, or exclude `date > today` from the
  elapsed-month stats.
- **CSV formula injection** (`export.dart:12-21`): quoting doesn't neutralize
  leading `=`/`+`/`-`/`@` in category names/notes, and the file is explicitly
  meant for Excel. Self-inflicted only (the user's own data on their own
  machine) → LOW. Fix sketch (S): prefix a `'` when a field starts with a
  formula trigger.
- **Migrations not wrapped in an explicit transaction**
  (`database.dart:466-498`): a mid-upgrade crash could strand a DB between
  versions (MED confidence — drift's per-statement behavior). Fix sketch (S):
  wrap the `onUpgrade` body in `transaction(...)`; verify against the
  existing v4→v6 migration test.
- **Money stored as IEEE-754 REAL with no write-boundary rounding**
  (`tables.dart` RealColumns): accepted for now — display rounds to 2dp and
  drift stays sub-cent at personal scale; an integer-cents migration is L
  effort/HIGH risk. Cheap mitigation if ever touched: round to 2dp in DAO
  add methods.
- **`restoreBackupJson` surfaces raw TypeErrors for malformed rows**
  (`backup.dart:154-239`): atomicity holds (transaction rolls back), so this
  is UX not data-safety; plan 016's generic catch covers the user-facing
  side. Tighten to `BackupFormatException` per-row if the format ever goes
  public.

## Findings considered and rejected

Carried over from batches 1–2 (do not re-audit):

- Uncapped transaction loads (dashboard/weekly budget): by design.
- Guarding `catchUp()` in `main()`: loud crash is the chosen failure mode.
- `flutter_local_notifications` 17→19: deferred, no pressure.
- Stale pubspec comment (removed chime dep): trivial.
- Async-gap `BuildContext` in add_transaction/settings: vetted clean.
- `_monthlyLabel` double-compute; `_skippedDeposits` unbounded prefs growth:
  trivia.

New this round:

- **Nested `StreamBuilder`s re-created in `build()`** (home/statistics):
  drift's stream-query store caches identical queries and home rebuilds are
  rare (resume/theme change) — measured harm ≈ none. Rejected.
- **Resume handler "races" `catchUp()` against `refreshAlerts()`**
  (`home_screen.dart:50-52`): both are fire-and-forget, but any transaction
  the catch-up inserts triggers stream refreshes, and the next resume
  re-arms alerts — staleness is bounded and self-correcting. Rejected
  (the *mutation-site* refresh gaps are the real issue → plan 015).
- **`monthsUntilDeadline` ignores day-of-month**: matches its documented
  spec and tests. Rejected.
- **`RecurrenceEngine._running` check-then-act**: no await between check and
  set; safe under Dart's single-threaded event loop. Rejected.
- **Landscape orientations enabled on iPhone** (Info.plist): cosmetic,
  no evidence of breakage. Not worth a plan.

## Direction — where to take the app next (grounded options, maintainer's call)

1. **Ship backup & restore** → planned as **016** (engine + tests already
   exist from the 004 spike; it also fixes the export-without-import
   asymmetry and completes 009's story). *Recommended first.*
2. **Cash in on category budgets**: over-budget alert (a fifth notification
   id fed from `budgetProgress` inside `refreshAlerts`) and a home-screen
   budget strip. The v6 scaffolding (`BudgetLine.isOver`,
   `alerts_coordinator`) makes both cheap. M.
3. **Transaction search**: notes are captured and rendered everywhere but
   findable nowhere; month browsing is the only history access. A search
   field over `watchAllWithCategory` (Dart-side filter is fine at this
   scale). M. Pairs with **transaction editing** (see unplanned list).
4. **Per-occurrence recurrence exceptions**: README defers it deliberately,
   and `transactions.recurrenceId` was added for exactly this. Skip/edit one
   occurrence without touching the rule. L — spike first.
5. **Currency setting**: 'ر.س' is hardcoded at ~25 sites, narrowing an
   Arabic-first app to Saudi users; Egypt/UAE/Kuwait/Jordan users see the
   wrong currency on every number. S–M (one setting + the money-format
   helper consolidation above).
6. **CSV export share sheet**: the store listing promises CSV export, but on
   iOS the file is written into an app-sandbox directory the user cannot
   reach — effectively broken for the whole iOS install base. Once 016 adds
   `share_plus`, wiring the two export tiles through the share sheet is S.
   (Treat as the first follow-up to 016.)
7. **Hijri-anchored recurrence** (weakest evidence, mentioned for
   completeness): display supports Hijri, but rules are Gregorian-only —
   salary/zakat tied to Hijri months can't be modeled. The `hijri` dep is
   already present. M–L; validate demand first.
